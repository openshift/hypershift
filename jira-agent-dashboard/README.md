# JIRA Agent Dashboard

A web dashboard and data pipeline for tracking [jira-agent](https://github.com/openshift/jira-agent) performance — merge rates, API costs, review comment quality, and PR lifecycle metrics.

## Architecture

```
┌─────────────────────────────────────────────────────────┐
│  OpenShift Cluster (jira-agent-dashboard namespace)     │
│                                                         │
│  ┌──────────────┐     ┌──────────────────────────────┐  │
│  │  Route (TLS) │────▶│  oauth-proxy  (:4443)        │  │
│  │  reencrypt   │     │  ┌────────────────────────┐  │  │
│  └──────────────┘     │  │  dashboard   (:8080)   │  │  │
│                       │  │  REST API + static web │  │  │
│                       │  └──────────┬─────────────┘  │  │
│                       └─────────────┼────────────────┘  │
│                                     │                   │
│                              ┌──────▼──────┐            │
│                              │  SQLite DB  │            │
│                              │  (PVC 1Gi)  │            │
│                              └──────▲──────┘            │
│                                     │                   │
│  ┌──────────────────────────────────┼────────────────┐  │
│  │  CronJobs (daily @ 06:00/06:10 UTC)               │  │
│  │                                                    │  │
│  │  scraper-prow ──▶ GCS artifacts (Prow job results)│  │
│  │  scraper-github ▶ GitHub API (PRs, comments)      │  │
│  └────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### Components

| Component | Type | Description |
|-----------|------|-------------|
| **dashboard** | Deployment | REST API server + vanilla HTML/JS/CSS frontend with Chart.js |
| **oauth-proxy** | Sidecar | OpenShift OAuth proxy — requires cluster login to access |
| **scraper-prow** | CronJob | Scrapes GCS for Prow job artifacts (daily 06:00 UTC) |
| **scraper-github** | CronJob | Fetches PR metadata and review comments from GitHub API (daily 06:10 UTC) |
| **SQLite DB** | PVC | WAL-mode database shared between dashboard and scrapers |

### Database Schema

5 tables in SQLite (WAL mode for concurrent readers):

- **job_runs** — Prow job execution records (build ID, timestamps, artifact URL)
- **issues** — JIRA issues with linked PR data (state, dates, merge duration)
- **phase_metrics** — Claude API usage per phase (tokens, cost, duration, model)
- **review_comments** — GitHub PR review comments with severity/topic classification
- **pr_complexity** — Lines added/deleted, files changed, complexity deltas

### Frontend Pages

| Page | Description |
|------|-------------|
| **Overview** | Summary cards (total issues, merge rate, cost), status/cost/duration charts, top reviewers, comment severity, recent activity feed |
| **Issues** | Sortable table of all issues with PR status, dates, lines changed, cost |
| **Comments** | Filterable comment list with severity/topic charts, pattern table, markdown report download |
| **Issue Detail** | Per-issue breakdown with phase metrics, cost, and individual comment classification |

## Prerequisites

- OpenShift cluster (tested on 4.x HostedClusters)
- `podman` for container builds (use `--platform linux/amd64` on ARM Macs)
- GitHub App with read access to the target repository (for scraper)
- `kubectl` / `oc` with cluster-admin or namespace-admin access

## Secrets Setup

Three secrets are required. These are **not** managed by kustomize to prevent accidental credential commits.

### 1. GitHub App Credentials

```bash
cp deploy/secrets.yaml.example deploy/secrets.yaml
# Edit deploy/secrets.yaml with your GitHub App ID and Installation ID
kubectl apply -f deploy/secrets.yaml
```

### 2. GitHub App Private Key

```bash
kubectl -n jira-agent-dashboard create secret generic github-app-key \
  --from-file=private-key.pem=/path/to/your-app.private-key.pem
```

### 3. OAuth Proxy Session Secret

```bash
openssl rand -base64 32 > /tmp/session_secret
kubectl -n jira-agent-dashboard create secret generic dashboard-proxy-session \
  --from-file=session_secret=/tmp/session_secret
rm /tmp/session_secret
```

> **Note:** The TLS certificate (`dashboard-tls`) is auto-generated by OpenShift via the `service.beta.openshift.io/serving-cert-secret-name` annotation on the Service.

```bash

# 3. Verify
kubectl -n jira-agent-dashboard get pods
kubectl -n jira-agent-dashboard get route dashboard
```

### What Gets Created

| Resource | Name | Description |
|----------|------|-------------|
| Namespace | `jira-agent-dashboard` | Isolated namespace |
| Deployment | `dashboard` | Dashboard pod (oauth-proxy + dashboard containers) |
| Service | `dashboard` | Exposes ports 4443 (https) and 8080 (http) |
| Route | `dashboard` | TLS reencrypt route to oauth-proxy |
| PVC | `dashboard-db` | 1Gi volume for SQLite database |
| CronJob | `scraper-prow` | Daily Prow artifact scraper |
| CronJob | `scraper-github` | Daily GitHub API scraper |
| NetworkPolicy | `dashboard-allow-ingress` | Only allows traffic from OpenShift router |
| NetworkPolicy | `deny-all-scraper` | Blocks all ingress to scraper pods |
| ClusterRoleBinding | `dashboard-auth-delegator` | Allows oauth-proxy to validate tokens |
| ServiceAccount | `dashboard` | With OAuth redirect annotation |

## Restore from Scratch

Complete checklist to rebuild on a new cluster:

```bash
# 1. Set kubeconfig
export KUBECONFIG=/path/to/cluster-kubeconfig

# 2. Build and push image
podman build --platform linux/amd64 -t quay.io/rh_ee_brcox/jira-agent-dashboard:latest .
podman push quay.io/rh_ee_brcox/jira-agent-dashboard:latest

# 3. Deploy namespace + resources (creates namespace, PVC, deployment, route, etc.)
kubectl apply -k deploy/

# 4. Create secrets (NOT in kustomize — do these manually)
#    a. GitHub App credentials
cp deploy/secrets.yaml.example deploy/secrets.yaml
# Edit with real values
kubectl apply -f deploy/secrets.yaml

#    b. GitHub App private key
kubectl -n jira-agent-dashboard create secret generic github-app-key \
  --from-file=private-key.pem=/path/to/your-app.private-key.pem

#    c. OAuth proxy session secret
openssl rand -base64 32 > /tmp/session_secret
kubectl -n jira-agent-dashboard create secret generic dashboard-proxy-session \
  --from-file=session_secret=/tmp/session_secret
rm /tmp/session_secret

# 5. Wait for pods to start
kubectl -n jira-agent-dashboard rollout status deployment/dashboard

# 6. Get the dashboard URL
kubectl -n jira-agent-dashboard get route dashboard -o jsonpath='{.spec.host}'

# 7. (Optional) Trigger an initial scrape instead of waiting for the CronJob
kubectl -n jira-agent-dashboard create job scraper-prow-init --from=cronjob/scraper-prow
# Wait for it to complete, then:
kubectl -n jira-agent-dashboard create job scraper-github-init --from=cronjob/scraper-github
```

> **Data note:** The SQLite database lives on the PVC. If the PVC is lost, the scrapers will rebuild the data on their next run. No manual data restore is needed — just re-run the scraper jobs.

## API Reference

All endpoints require authentication via the oauth-proxy. The dashboard container listens on `:8080` (internal only).

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/issues?from=YYYY-MM-DD&to=YYYY-MM-DD` | List issues in date range |
| `GET` | `/api/issues/{id}` | Issue detail with phases and comments |
| `GET` | `/api/comments/summary?from=YYYY-MM-DD&to=YYYY-MM-DD` | All comments in date range |
| `GET` | `/api/comments/{issueID}` | Comments for a specific issue |
| `PATCH` | `/api/comments/{id}` | Update comment classification (severity + topic) |
| `GET` | `/api/trends?from=YYYY-MM-DD&to=YYYY-MM-DD` | Weekly trend aggregates |
| `GET` | `/healthz` | Health check (returns `ok`) |

### Classification Values

**Severity:** `nitpick`, `suggestion`, `required_change`, `question`, `unclassified`

**Topic:** `style`, `logic_bug`, `test_gap`, `api_design`, `documentation`, `ci`, `approval`, `process`, `unclassified`

## Security

| Layer | Implementation |
|-------|---------------|
| **Authentication** | OpenShift OAuth proxy sidecar — users must log in via cluster OAuth |
| **Authorization** | `openshift-delegate-urls` — requires `get` permission on `services` in the namespace |
| **TLS** | Route uses `reencrypt` termination; service-serving-cert for pod TLS |
| **Network** | NetworkPolicy restricts dashboard ingress to OpenShift router; scrapers deny all ingress |
| **Containers** | Non-root (`USER 1001`), `readOnlyRootFilesystem`, all capabilities dropped |
| **Input validation** | PATCH body limited to 1KB, severity/topic validated against allowlists |
| **XSS prevention** | All `innerHTML` interpolation uses `escapeHTML()` |
| **SQL injection** | All queries use parameterized placeholders |
| **Error handling** | Internal errors logged server-side; generic messages returned to client |
| **Secrets** | Excluded from kustomization and `.gitignore`; private key mounted at `0400` |
